# Hugo and Docsy Migration Plan

## Objective
Migrate the `FsFlow` documentation site from Docusaurus to Hugo using the Docsy theme. Docusaurus has been presenting stability, performance, and upgrade issues. The goal is to move to a more stable, faster build system while preserving the current documentation content, structure, and generation scripts. We will use the default Docsy visual theme.

## Context
- Current site is in `site/` using Docusaurus.
- Documentation source is in `docs/`.
- Build scripts (`scripts/build-docs-site.sh`) trigger API and example docs generation, then build the site.
- `site/sidebars.js` dictates the current Docusaurus navigation structure.

## Phase 1: Initialize Hugo and Docsy
1. **Tool Setup**: Update `mise.toml` to include `hugo = "extended"`.
2. **Skeleton Creation**: Replace the `site` directory with a fresh Hugo site initialization.
3. **Docsy Module**: Add Docsy as a Hugo module.
4. **Cleanup**: Remove Docusaurus configuration and dependencies: `package.json`, `package-lock.json`, `docusaurus.config.js`, `sidebars.js`, and Docusaurus-specific source folders like `src/`.

## Phase 2: Content and Navigation Migration
1. **Content Mount**: Configure Hugo to mount the `docs/` directory to Hugo's `content/` path, or move the contents of `docs/` directly into the Hugo site structure depending on what works best for the existing auto-generation scripts.
2. **Navigation Structure**: Docsy uses directory structures and `_index.md` files for its sidebar.
   - Translate `site/sidebars.js` into Hugo `_index.md` files for categories (`Start`, `Core Model`, `Patterns`, `Ecosystem`, and `Reference`).
   - Use frontmatter `weight` properties to preserve the explicit ordering currently defined in `sidebars.js`.
3. **Homepage**: Update `docs/index.md` (which currently serves as the Docusaurus homepage) to be compatible with Docsy's homepage layout (usually `_index.md` at the root).
4. **Frontmatter Updates**: Ensure existing markdown files have compatible frontmatter (`title`, `description`, `weight`).

## Phase 3: Build Scripts and CI
1. **Update `scripts/build-docs-site.sh`**:
   - Remove `npm ci` and `npm run build` commands for Docusaurus.
   - Replace with `hugo --source "$root_dir/site" --destination "$root_dir/output"` (matching the previous output directory configured in Docusaurus's package.json).
2. **Update CI Workflows**:
   - Ensure any GitHub Actions workflows building the docs use `hugo-extended` instead of Node/npm for the site build step.

## Phase 4: Validation
- Ensure `bash scripts/build-docs-site.sh` completes successfully and is fast.
- Verify that all API docs (generated by `scripts/generate-api-docs.mjs`) are properly included in the Docsy sidebar and formatted correctly.
- Verify that the runnable examples (`scripts/generate-example-docs.sh`) render correctly in the new theme.
- Confirm that the default Docsy visual theme looks acceptable without extensive custom CSS.
